## 🎯 系统概述

自适应记忆系统采用**分层记忆架构**，将记忆分为：

- **短期记忆（STM）**：存储在 SQLite 中，用于保存当前会话的上下文
- **长期记忆（LTM）**：通过 LLM 总结和向量化后存储在 Milvus 和 SQLite 中

### 数据流转过程

```
用户消息 → STM (SQLite) 
    ↓
AI 回复 → STM (SQLite)
    ↓
[达到阈值/手动触发]
    ↓
LLM 总结提取 → 向量化 → LTM (Milvus + SQLite)
```

## 📚 核心概念

### 1. 会话（Session）

一个会话代表一次完整的对话交互，包含：
- `session_id`: 唯一标识符
- `user_id`: 用户 ID
- `agent_id`: 智能体 ID
- `session_type`: 会话类型（conversation/task/query）
- `context_length`: 当前上下文长度
- `max_context_length`: 最大上下文长度

### 2. 消息（Message）

会话中的单条消息，包含：
- `message_id`: 唯一标识符
- `session_id`: 所属会话 ID
- `role`: 角色（user/assistant）
- `content`: 消息内容
- `token_count`: Token 数量（可选）
- `importance_score`: 重要性评分（可选）

### 3. 记忆提取

将短期记忆转换为长期记忆的过程：
1. **收集**：从 STM 中获取会话的所有消息
2. **合并**：将消息合并为完整内容
3. **总结**：使用 LLM 进行总结和结构化提取
4. **向量化**：生成向量嵌入
5. **存储**：保存到 Milvus（向量）和 SQLite（元数据）

## 🛠️ 实现方案

### 方案一：使用现有 API（推荐）

系统已提供完整的 API，可直接使用。

#### 写入会话历史

**API 端点：** `POST /api/v1/memory/storage/stm`

**请求体：**
```json
{
  "userId": "user_123",
  "agentId": "agent_456",
  "sessionType": "conversation",
  "role": "user",
  "content": "用户的消息内容",
  "maxContextLength": 4096,
  "retentionHours": 24
}
```

**响应：**
```json
{
  "sessionId": "01ARZ3NDEKTSV4RRFFQ69G5FAV",
  "messageId": "01ARZ3NDEKTSV4RRFFQ69G5FAW"
}
```

#### 提取长期记忆

**方式 A：手动触发转移**

**API 端点：** `POST /api/v1/memory/storage/transfer`

**请求体：**
```json
{
  "sessionId": "01ARZ3NDEKTSV4RRFFQ69G5FAV",
  "messageCountThreshold": 100
}
```

**响应：**
```json
{
  "entryIds": ["entry_1", "entry_2"]
}
```

**方式 B：直接存储为长期记忆**

**API 端点：** `POST /api/v1/memory/storage/ltm`

**请求体：**
```json
{
  "sourceId": "session_123",
  "sourceType": "conversation",
  "content": "完整的会话内容",
  "title": "会话标题（可选）"
}
```

**响应：**
```json
{
  "entryId": "01ARZ3NDEKTSV4RRFFQ69G5FAX"
}
```

### 方案二：自动转移服务

系统已实现自动转移服务，会定期检查并自动转移符合条件的会话。

**初始化（在 main.rs 中）：**
```rust
use crate::services::memory_transfer::init_transfer_service;

// 参数：检查间隔(秒), 消息数量阈值, 会话时间阈值(小时)
```

## 💻 代码示例

### 示例 1：完整的对话处理流程

```rust
use crate::services::memory_storage::MemoryStorageService;
use crate::db::stm::STMRepository;

async fn handle_conversation(
    user_id: &str,
    agent_id: &str,
    user_message: &str,
) -> Result<()> {
    // 1. 写入用户消息到 STM
    let (session_id, _) = MemoryStorageService::store_stm(
        user_id,
        agent_id,
        "conversation",
        "user",
        user_message,
        4096,
        24,
    )
    .await?;
    
    // 2. 生成 AI 回复（这里需要调用你的 LLM 服务）
    let ai_response = generate_ai_response(user_message).await?;
    
    // 3. 写入 AI 回复到 STM
    MemoryStorageService::store_stm(
        user_id,
        agent_id,
        "conversation",
        "assistant",
        &ai_response,
        4096,
        24,
    )
    .await?;
    
    // 4. 检查是否需要转移到 LTM
    let messages = STMRepository::get_session_messages(&session_id, None).await?;
    if messages.len() >= 100 {
        // 自动转移到 LTM
        MemoryStorageService::auto_transfer_stm_to_ltm(&session_id, 100).await?;
    }
    
    Ok(())
}
```

### 示例 2：批量写入会话历史

```rust
async fn batch_write_conversation(
    user_id: &str,
    agent_id: &str,
    messages: Vec<(&str, &str)>, // (role, content)
) -> Result<String> {
    // 创建会话
    let session_id = STMRepository::create_session(
        user_id,
        agent_id,
        "conversation",
        4096,
        24,
    )
    .await?;
    
    // 批量添加消息
    for (role, content) in messages {
        STMRepository::add_message(
            &session_id,
            role,
            content,
            None,
            None,
        )
        .await?;
    }
    
    Ok(session_id)
}
```

### 示例 3：智能记忆提取

```rust
async fn smart_extract_memory(session_id: &str) -> Result<()> {
    // 1. 获取会话消息
    let messages = STMRepository::get_session_messages(session_id, None).await?;
    
    // 2. 计算重要性评分（可以根据消息内容、长度等计算）
    let total_importance: f64 = messages
        .iter()
        .map(|m| m.importance_score.unwrap_or(0.5))
        .sum();
    
    // 3. 如果重要性足够高，提取为长期记忆
    if total_importance > 10.0 {
        MemoryStorageService::auto_transfer_stm_to_ltm(session_id, 50).await?;
    }
    
    Ok(())
}
```

## ✨ 最佳实践

### 1. 会话管理

- **复用会话 ID**：同一对话使用相同的 `session_id`，系统会自动关联消息
- **合理设置保留时间**：根据重要性设置 `retention_hours`
  - 重要对话：72 小时
  - 普通对话：24 小时
  - 临时对话：1 小时

### 2. 记忆提取策略

- **消息数量阈值**：
  - 小型对话：50 条消息
  - 中型对话：100 条消息
  - 大型对话：200 条消息

- **时间阈值**：
  - 快速提取：12 小时
  - 标准提取：24 小时
  - 延迟提取：72 小时

- **重要性评分**：为重要消息设置 `importance_score`，优先提取

### 3. 性能优化

- **异步处理**：记忆提取是耗时操作，建议异步执行
- **批量操作**：使用批量 API 减少数据库交互
- **缓存策略**：对于频繁访问的会话，可以添加缓存

### 4. 错误处理

```rust
match MemoryStorageService::store_stm(...).await {
    Ok((session_id, message_id)) => {
        // 处理成功
    }
    Err(e) => {
        // 记录错误日志
        tracing::error!("Failed to store STM: {}", e);
        // 可以重试或使用备用方案
    }
}
```

## ❓ 常见问题

### Q1: 如何判断会话是否需要提取为长期记忆？

**A:** 可以根据以下条件判断：
- 消息数量达到阈值（如 100 条）
- 会话时间超过阈值（如 24 小时）
- 重要性评分累计达到阈值
- 用户手动触发

### Q2: 记忆提取失败怎么办？

**A:** 检查以下几点：
1. LLM 服务是否正常运行（检查 Ollama）
2. 嵌入模型服务是否正常
3. Milvus 连接是否正常
4. 查看日志中的具体错误信息

### Q3: 如何避免重复提取？

**A:** 系统会在转移后标记会话状态，但建议：
- 在转移前检查会话是否已转移
- 使用 `session_id` 作为 `source_id` 存储到 LTM，便于去重

### Q4: 如何提高记忆提取的质量？

**A:** 
1. 为重要消息设置 `importance_score`
2. 在提取前过滤无关消息
3. 调整 LLM 的提示词以获得更好的总结
4. 使用批量提取时，按主题分组

### Q5: 如何处理大量历史数据？

**A:**
1. 使用批量 API (`batch_store_ltm`)
2. 分批处理，避免一次性处理过多数据
3. 使用异步任务队列
4. 监控系统资源使用情况
