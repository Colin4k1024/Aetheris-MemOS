# 为什么使用 Axum

HTTP 层迁移说明，以及它如何与开源路线图对齐。

---

## 当前状态：Axum

后端现已使用 **Axum**（Rust）实现路由、提取器、中间件和服务启动。

- **为何改为 Axum：** Rust Web 生态更主流、示例更丰富、开源贡献者上手成本更低。
- **兼容策略：** 对外接口保持稳定（`/api/*`、`/scalar`、`/api-doc/openapi.json`），内部实现统一为 Axum。

---

## 迁移原则

| 原则       | 决策                                              |
| ---------- | ------------------------------------------------- |
| API 稳定性 | 保持公开端点与默认端口不变                        |
| 实现一致性 | 使用类型化提取器与统一 `AppError -> IntoResponse` |
| 中间件等价 | 保留 CORS、JWT、限流行为                          |
| 可维护性   | 保持服务层与领域逻辑尽量框架无关                  |

---

## 与路线图的关系

框架迁移已完成；后续重点是架构能力建设（决策追踪持久化、可观测性、运行时集成），不再围绕框架切换。
